## Guidelines

### What Deserves Its Own Model

**Create a model for:** people, physical locations, assets, machines, external services/vendors.

**Do NOT create a model for:** financial metrics, abstract concepts, or derived values — these are stats on a parent model.

---

### Spatial Layout

Think in meters. Note when entities exist physically inside or on top of another. Arrange unrelated entities in a logical grid with enough space that they don't visually overlap. Timeline markers sit below the models on the y-axis, centered on x.

Rough footprint defaults:
- Large building: ~10–15m wide
- Medium structure (pond, pen): ~5–8m wide
- Small entity (person, animal): ~1–2m wide

---

### Choosing a Time Unit

Pick the finest unit that keeps the simulation meaningful:

| Scenario | Suggested unit |
|---|---|
| Financial modeling, business growth | month or quarter |
| Retail operations, staffing | day or week |
| Animal behavior, foot traffic | minute or hour |
| Fast-moving (traffic, machinery) | second |

Every entity in the scene uses this same unit.

---

### Lifecycle Design

Use a lifecycle for any entity that ages, degrades, or expires. Don't assign one to pure value-holders (a fixed salary, a static cost).

A lifecycle has four phases — choose vocabulary that fits the entity's domain:

| Domain | Phase 0 | Phase 1 | Phase 2 | Terminal |
|---|---|---|---|---|
| Living thing | alive | aging | elderly | dead |
| Machine | active | degraded | critical | decommissioned |
| Business / asset | operating | declining | expiring | closed |

For each lifecycle entity, specify:
- **Lifecycle chain** — e.g. alive → aging → elderly → dead
- **max_age** — how many steps. Set 1.5–3× beyond the final marker so entities outlive the observable window slightly.
- **Phase thresholds** — Phase 1 at ~50–65% of max_age; Phase 2 at ~75–85%
- **spawn_tick** — 0 for entities born at start; negative for seed entities already partway through life. Stagger across same-type entities to spread deaths across the timeline.
- **Terminal behavior** — hide (default) or freeze
- **Speed scaling** — whether movement speed scales across phases (yes for most moving entities)

---

### Spawning Design

Use spawning for entities that can mate and produce offspring. Requires a lifecycle.

For each spawning entity, specify:
- **Partner** — which entity it mates with
- **Maturity threshold** — fraction of max_age before mating is possible (typically 10–20%)
- **Mating cooldown** — steps between matings (typically 15–25% of max_age)
- **Mating proximity** — distance in meters to trigger mating

Always declare spawning on both entities in a pair.

---

### Reactive Behavior

For every entity, explicitly describe all listeners — what triggers them and what they do. The builder has no other source of truth for this. Be specific about tag names and values.

**Listener naming** — always specify the exact listener name when describing reactive behavior:
- Stat listener: `on<StatName>Changed` — e.g. `oncash_balanceChanged`
- State listener: `<stateName><value>OnEnter` — e.g. `lifecycleagingOnEnter`, `shiftworkingOnEnter`
- Custom listener: any meaningful name
- `onDeltaOverwrite`: replaces all per-tick logic. When present, this listener fires on every tick instead of the built-in delta function. The current tick value is available as `that` inside the listener. Use this for any stat that needs conditional, derived, or cross-model update logic. All lifecycle transitions, speed scaling, movement, stat recalculation, and mating checks should live here rather than in a `current_tick` stat delta.

**Lifecycle listeners** — describe what happens on every phase transition:
- What color the entity becomes
- Whether visibility changes (hide on terminal, restore on phase 0 reset)
- Any stat or state changes that accompany the transition

**Stat listeners** — if a stat crossing a threshold should trigger something, say so. Example: "when cash_balance drops below 0, change operating_status state to 'insolvent'."

**State listeners** — if entering a state should change tags or trigger behavior, describe it. Example: "when shift state enters 'on_break', set color to yellow."

**Per-tick behavior** — describe what happens every step: which stats are recalculated, how movement is determined, what conditions gate it, and which other models are read. Example: "each step, read monthly_salary from Employee 1 and add it to total_expenses on the Coffee Shop."

---

### Timeline Marker Placement

Markers are scrub points — place 3–5 at steps where something meaningfully different is happening. Do not place one at step 0.

Do not cluster markers at small step values when lifecycles are long. A simulation with max_age 200 should have markers in the hundreds.

Process:
1. Determine total step range from the longest-lived entity's max_age
2. Identify key events: first maturity, first mating, first death, first child maturing, population peak, financial inflection
3. Place markers at or just before those events — each marker has a `step` (the tick number) and a `timeUnit`

---

### Example

**Input:** "A duck pond with 2 ducks of different ages."

**Plan:**

The scene uses **minutes** as its time unit.

**Pond** — centered at the scene origin, ~6m across. Tracks duck_count. Stationary, no lifecycle.

**Duck 1** — sits on top of the Pond. Lifecycle: alive → aging → elderly → dead. max_age: 200 steps. Phase 1 at step 120 (60%), Phase 2 at step 160 (80%). spawn_tick: 0. Terminal: hide. Speed scales with phase. Wanders within the Pond bounds. Mates with Duck 2 — maturity threshold 15%, mating cooldown 40 steps, proximity 1.5m.
Reactive behavior:
- lifecyclealiveOnEnter: color white, restore visibility
- lifecycleagingOnEnter: color tan (#c8a882), speed 75% of base
- lifecycleelderlyOnEnter: color grey (#a0a0a0), speed 40% of base
- lifecycledeadOnEnter: hide the bot
- onDeltaOverwrite: tick value available as `that`. If lifecycle is not dead, wander within Pond bounds; check proximity to Duck 2 and shout mergeModels if mating conditions are met. Handle lifecycle transitions and speed scaling here too.

**Duck 2** — sits on top of the Pond. Same lifecycle and reactive behavior as Duck 1. spawn_tick: -30 (already 30 steps old at start, already mature, will die before Duck 1). Mates with Duck 1 on the same terms.

**Markers** — key events:
- step 30 (timeUnit: minute): Duck 1 reaches maturity, first mating window opens
- step 80 (timeUnit: minute): First offspring likely born, Duck 2 approaching Phase 1
- step 130 (timeUnit: minute): Duck 2 enters Phase 2, first child nearing maturity
- step 150 (timeUnit: minute): Duck 2 likely dead, second generation active
